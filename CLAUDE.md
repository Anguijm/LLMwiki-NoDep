# CLAUDE.md

Project operating manual for Claude Code sessions on LLMwiki-NoDep. Auto-loaded at session start.

## Prime directive: plan-first, human-approved

No autonomous coding without human sign-off. Every non-trivial change follows:

1. **Plan** — write the plan to `.harness/active_plan.md` (create or overwrite), commit, push.
2. **Council** — open (or push to) a PR so `.github/workflows/council.yml` runs the Gemini council against the PR diff, or run `council.py --plan` locally as a fallback.
3. **Approve** — surface the Lead Architect synthesis to the human; wait for explicit approval.
4. **Execute** — implement in small, testable increments on the same PR (each push re-runs council for diff review).
5. **Reflect** — append a KEEP / IMPROVE / INSIGHT / COUNCIL block to `.harness/learnings.md`.

Trivial changes that skip the council: typo fixes, single-line bug fixes, comment edits, reverting a failed change. Everything else routes through council.

## What counts as "approval" (hard rule, no exceptions)

An approval is **only** valid if all three are true:

1. The plan exists as a committed file at `.harness/active_plan.md` (tracked in git, pushed to origin — not just present in the agent's working tree).
2. The council has produced a synthesis covering the current plan via one of these paths:
   - **PR-triggered (primary):** the branch holding the plan has an open PR, and the most recent `<!-- council-report -->` comment on that PR was posted by `.github/workflows/council.yml` against a commit SHA at or ahead of the commit that last modified `.harness/active_plan.md`. A council report from a prior plan revision does NOT satisfy the gate — push the plan, let the workflow re-run, and confirm the marker comment was updated for the new SHA.
   - **Local fallback:** `.harness/last_council.md` was generated by `council.py --plan` against the tracked plan file and is newer than `.harness/active_plan.md` (requires `GEMINI_API_KEY` locally).
   - **Override:** the human wrote the literal phrase `override council: <reason>` in chat. Reserved for council-infrastructure bootstrapping or genuine emergencies, not routine work.
3. The human typed an explicit `approved`, `ship it`, `proceed`, or equivalent after seeing (1) and (2).

**Anti-pattern (forbidden):** Writing a plan, summarizing it in chat, getting "approved," and executing — without the plan being committed and without the council ever running. This discipline was forged in the sibling repo (LLMwiki_StudyGroup, 2026-04-16 incident — see `.harness/learnings.md`). Do not relitigate it here.

**Before writing any implementation code**, the agent MUST:
- `git ls-files --error-unmatch .harness/active_plan.md` succeeds AND `git status --porcelain .harness/active_plan.md` is empty (plan is tracked and has no unstaged edits).
- One of: (a) an open PR exists for the current branch and its latest `<!-- council-report -->` comment was posted against a commit SHA ≥ the commit that last modified the plan; (b) `.harness/last_council.md` exists locally and is newer than `active_plan.md`; (c) the human's most recent message contains `override council:`.
- None of these gates can be satisfied by the agent alone.

If either gate fails, the agent STOPS, surfaces the gap, and waits.

## Circuit breaker

If `.harness_halt` exists at repo root, stop all work. Print the file's contents to the user and do nothing else until the file is removed. See `.harness/halt_instructions.md`.

## Session I/O protocol

**At session start:**
- Read `.harness/session_state.json` to learn the active plan, last council run, current focus area, last commit.
- Read the last ~20 lines of `.harness/yolo_log.jsonl` to understand recent events.
- Skim `.harness/learnings.md` for accumulated context — especially the most recent COUNCIL and INSIGHT entries.

**At session end (or after each meaningful task):**
- Append a reflection block to `.harness/learnings.md`:
  ```
  ## <YYYY-MM-DD HH:MM> — <task title>
  ### KEEP
  - <what worked>
  ### IMPROVE
  - <what to change next time>
  ### INSIGHT
  - <non-obvious thing worth remembering>
  ### COUNCIL
  - <notable feedback from the council, if run>
  ```
- Update `.harness/session_state.json` (`active_plan`, `focus_area`, `last_council` as applicable).
- `.harness/yolo_log.jsonl` is append-only; add an event line describing the outcome.

The `post-commit` git hook handles `last_commit` bookkeeping automatically — do not edit that field by hand.

## When to run the council

**Primary path (PR-triggered):** `.github/workflows/council.yml` runs automatically on `pull_request` (opened / synchronize / reopened). It invokes `council.py --diff` against the PR base, generates the Lead Architect synthesis, and posts (or edits in place) a single `<!-- council-report -->` comment on the PR. No local secret required — the workflow uses the `GEMINI_API_KEY` Actions secret. This is the default flow: write the plan → commit → push → open a PR → wait for the council comment.

**Local fallback:** before opening a PR, run `python3 .harness/scripts/council.py --plan .harness/active_plan.md` against the committed plan. Requires `GEMINI_API_KEY` in the shell. Produces `.harness/last_council.md` (not committed). For diff-review of local working-tree changes: `council.py --diff`.

**Council is required** (either path) before writing code when any of these apply:

- New feature (anything user-facing or adding a new route in `index.html`, a new prompt template under `/_prompts/`, a new CSV column in `srs.csv`, a new tier folder, or a new note-format convention).
- Schema change — any non-backward-compatible edit to the frontmatter schema, `srs.csv` headers, or the `_index.md` regeneration contract.
- New runtime dependency (should be approximately never — NoDep's whole point is zero runtime deps; ask before adding anything that is not dev-only).
- New dev-time dependency — linter, test runner, or any addition to a future `devDependencies` block.
- New dev-time external API call (Gemini council is the only one today; any future dev-time AI integration routes through council).
- Change that crosses the `file://` → DOM boundary: the inline markdown parser, the `[[wiki link]]` resolver, the CSV parser, the File System Access API handle logic, or anything that turns on-disk content into DOM.
- Change to the File System Access API permission scope (what the user is asked to grant, and what the app reads/writes once granted).
- New Power Automate flow documented under `/power-automate/` (governance surface; the flows run on Microsoft infrastructure on behalf of the user).

**Council is also required for institutional-knowledge content** — files that future sessions read as ground truth on startup or load into every council review. Specifically:

- Any append to `.harness/learnings.md`. KEEP / IMPROVE / INSIGHT / COUNCIL entries compound across sessions; a single wrong claim in an INSIGHT block warps downstream decisions for as long as the file is read. Diff size does not determine review need; downstream leverage does.
- Any edit to `.harness/council/*.md` persona files. Changing a persona changes the lens every future council review uses — a meta-change with compounding blast radius.
- Any edit to `CLAUDE.md` itself (this file). Every session reads it at startup; wording changes are behavior changes.

Mechanical session-state updates — `.harness/session_state.json` pointers and `.harness/yolo_log.jsonl` event appends — are factual bookkeeping (pointing at SHAs, timestamps, PR numbers) and remain council-exempt. If a `session_state.json` change includes narrative `notes` text that is doing the work of a learnings entry, route it through council instead.

**Anti-pattern (forbidden, ported from sibling repo 2026-04-20):** Marking a session-reflection PR `[skip council]` because "the diff is small" or "it's just documentation." The `[skip council]` bypass is reserved for truly mechanical changes from the skip list (typo fixes, single-line bug fixes, comment edits, reverting a failed change) — not for content that future agents will treat as load-bearing truth.

The council reads the personas in `.harness/council/` (one per axis, excluding `lead-architect.md` and `README.md`) and the Lead Architect synthesizes a single authoritative plan. Present the synthesis to the human before executing.

**Cost caps:**
- Per-run: 15 Gemini calls, enforced by `council.py` (`CALL_CAP`).
- Per-month: `MONTHLY_CAP` in `.github/workflows/council.yml` (cache-based counter, outside PR blast radius).
- `[skip council]` in a PR title (case-insensitive) bypasses the PR-triggered run for that push.

## Non-negotiables (no council run can override these)

- **Never assign untrusted content to `.innerHTML`.** The inline markdown parser, wiki-link resolver, and any other on-disk-content → DOM path must escape by default. This is the vanilla-JS analog of `dangerouslySetInnerHTML`: the same rule applies, stricter — there's no React reconciliation buffer.
- **No PII or API keys in logs.** `console.log` / `console.error` redact before serialization. SharePoint-sync'd files are visible to anyone with folder access; log payloads must assume that audience.
- **No raw ingested content passed to `eval`, `new Function`, or `setTimeout(string, ...)`.** Ever.
- **Runtime deps stay at zero.** No `<script src="https://...">`, no CDN fonts, no `import` from `node_modules`, no network calls from `index.html`. The app must work with the laptop's Wi-Fi off.
- **Conventional commits** (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`). See `CONTRIBUTING.md`.

## Cost posture

Runtime API cost: **$0/month.** There is no runtime AI. All "AI" is a human copy/paste loop against GenAI.mil; the model time is the human's time, not ours.

Discipline: each `/_prompts/*.md` template documents its **expected human-turn budget on GenAI.mil** in its frontmatter (how many paste round-trips it assumes before a usable output). A template that routinely requires 4+ round-trips to converge is a Cost-council target. Redundant templates (two templates that overlap in purpose) are also Cost-council targets.

Dev-time cost:
- **Gemini 2.5 Pro** — council reviews only (PR-triggered CI action + local dev fallback). Hard cap: 15 calls per run, ~60 runs per month. ~$1–3/month.
- No other dev-time AI calls without council approval.

## Stack invariants

- **Vanilla HTML + ES2022 JavaScript.** No TypeScript at runtime, no JSX, no framework.
- **No build step.** `index.html` is shipped as-is; it is what the user opens in Edge.
- **Inline CSS + inline `<script>` in `index.html`.** No bundler, no asset pipeline.
- **File System Access API** (primary) with a **drag-drop folder fallback** for when the API is unavailable or permission is denied.
- **Storage layer = SharePoint-synced folder.** The repo's working directory is the app's live filesystem. There is no server, no database, no cloud storage distinct from SharePoint.
- **"AI layer" = manual GenAI.mil copy/paste.** The `/_prompts/` templates are what the user pastes; GenAI.mil's output is what the user pastes back.
- **Three-tier memory via folder convention:** `/bedrock/`, `/warm/`, `/cold/` — tier selected per note via frontmatter `tier:` field.
- **SRS via `srs.csv` + SM-2.** Simple algorithm, portable file format, human-inspectable. Written back via the File System Access API when the user completes a review session.

## Runtime environment constraints

These are the target deployment reality. Future council reviews measure plans against them.

- **Opened via `file://`** in Microsoft Edge on a locked-down work laptop. No localhost server, no module bundler, no `import` from `node_modules` at runtime.
- **SharePoint-synced folder.** Files may briefly disappear during sync; two devices editing the same file is last-write-wins (no merge UI). The "filesystem" is eventually consistent, not strictly local.
- **No network egress** from the app itself. Any feature that needs external data is a human-mediated copy/paste step, not a `fetch()`.
- **No telemetry, no analytics, no error reporting.** A thrown exception surfaces to `console.error` and nowhere else.
- **Dev-time Node tooling is allowed** (tests, linter, local preview server) under a `devDependencies` block, but the shipped folder must never require them to run.

A plan that breaks any of these constraints is architecturally unsound for this repo, regardless of how clean the code is.

## Before committing

1. `npm run lint` passes (once the Phase 2 dev `package.json` exists).
2. `npm run test` passes for touched modules (ditto).
3. `.harness/scripts/security_checklist.md` reviewed for any surface this change touches.
4. **Open `index.html` from `file://` in Edge (or a chromium-based browser) and exercise the touched feature manually.** Confirm the golden path works; spot-check one edge case. Type checking and unit tests verify code correctness, not feature correctness.
5. Conventional-commit message.

## Hook timeouts (`.claude/settings.json`)

JSON doesn't allow inline comments, so the rationale for the two configured hook timeouts lives here:

- **`SessionStart` hook timeout: 10s.** Runs `.claude/hooks/session-start.sh`, which prints last-commit / halt-status / active-plan / last-council-verdict at session start. Typical run < 2s (just local file reads + `git log`). **Failure mode if exceeded:** the hook is killed and Claude proceeds without the session-start context — the cost is missing the halt warning or active-plan reminder, which is annoying but not unsafe. If you find this timing out regularly, optimize the hook script rather than raising the timeout.
- **`PreToolUse` (Bash) hook timeout: 15s.** Runs `.claude/hooks/check-branch-not-merged.sh` before any Bash tool call. The check involves `git fetch origin main`, which can be slow on poor networks. **Failure mode if exceeded:** the hook fails OPEN and the push is allowed. **This is by design** — `check-branch-not-merged.sh` is a **best-effort developer guardrail, not a security gate**. The hard merge-discipline gates are PR review + the council + `branch-guard.yml`. The pre-push hook just catches the common "I'm on a stale branch" mistake earlier than the council would. If you need stricter behavior, swap to fail-closed in the hook itself, but be ready for hooks to refuse pushes on flaky networks.
