## Scope — six file additions in one PR

### Add 1 — `harness.yml` (config baseline)

Generated by `harness init --update`. Declares stack, council caps (already enforced separately), and three new opt-in command blocks for V0.3.0:

- `research:` — `model: gemini-2.5-pro`. Default cap is all active personas; a `max_personas: <n>` override is commented in but not active by default.
- `synthesize:` — `model: gemini-2.5-flash`, `max: 5`, `min_cluster_size: 3`.
- `recall:` — `embedding_model: text-embedding-004`, `vector_weight: 0.5`.

These are inert until a developer runs the corresponding command. There is no implicit cron, no implicit invocation. **All cost is opt-in per-invocation** (see Cost analysis below).

### Add 2 — `.github/workflows/drift-check.yml` (weekly topology check)

Read-only weekly workflow that clones harness-cli at the pinned SHA and verifies this repo's topology matches. On drift: opens (or updates) a single tracking issue with the report. Manual run via `workflow_dispatch`.

Pinned at:
```
HARNESS_VERSION: V0.3.0
HARNESS_SHA: 519a3f5fd1474c8f24d188b298c60bad1f62ca76
```

**SECURITY NOTE will be added inline** explaining:
- The supply-chain model: weekly clone of `Anguijm/harness-cli@<SHA>`. SHA pinning protects against tag force-push (the tag is the human-readable label; the SHA is the immutable cryptographic pin).
- The blast radius: drift-check executes harness-cli's `harness check` and reads template files. It does NOT execute consumer-repo code in the harness-cli runtime.
- The trust posture: harness-cli is maintained by the same human running this repo. Same trust level as the repo itself; not a third-party dependency.
- The mitigation: any future bump to `HARNESS_SHA` is a deliberate PR. A compromised tag with no SHA bump cannot reach this workflow.

**HARNESS_VERSION/HARNESS_SHA staleness mitigation** (per the bugs persona's R1 concern): the inline comment block on these env vars will explicitly say "if you bump one, bump the other; an out-of-sync pair will report a misleading version label in any drift-detected issue."

### Add 3 — `.claude/` (Claude Code session integration)

Four files: `hooks/session-start.sh`, `hooks/check-branch-not-merged.sh`, `settings.json`, `skills/close-session.md`.

These wire up the Claude Code session-management surface:

- **`session-start.sh`** prints last commit, halt status, active plan, last council verdict at session start. Magic numbers (head line counts) will be commented inline per the maintainability persona's R1 ask.
- **`check-branch-not-merged.sh`** is a PreToolUse Bash hook that refuses pushes from squash-merged branches. Has a 15s timeout because `git fetch origin main` can be slow on poor networks; failure is fail-open by design (see CLAUDE.md "Hook timeouts" section, which will gain its rationale comments since JSON can't carry inline comments).
- **`settings.json`** carries hook timeouts and permissions config. Magic-number rationale documented in CLAUDE.md alongside the hook descriptions.
- **`close-session.md`** is the 11-step session-close ritual including failure-mode audit and `harness lint` step.

These are all NoDep-compatible: no runtime impact, no AI calls during normal hook execution, no network egress. Hooks fire only inside Claude Code sessions.

### Add 4 — `.harness/council/maintainability.md`

The 7th canonical reviewer persona. The other 6 already exist in this repo's `.harness/council/`; maintainability was missing.

**Cost:** adds 1 Gemini Pro call per PR review. At ~10 PRs/month and ~$0.01/call (Pro pricing for ~5K-token persona prompt + ~500-token response), this is **+$0.10/month**.

**Why include it:** maintainability concerns (magic numbers, unclear timeouts, undocumented constants) recur across this repo's recent PRs (see `.harness/learnings.md`). A dedicated reviewer surfaces them at review time rather than as Codex P2 follow-ups.

### Add 5 — `scripts/setup-secrets.sh`

One-time bootstrap helper that pipes a local `.env`-style secrets file into `gh repo secret set`. Already secure-by-design (uses `gh` stdin piping, not URL or env exposure).

**Justification:** without this, the `GEMINI_API_KEY` setup for new clones / forks is a manual, error-prone process. The script is run once per environment; it does not run in CI.

### Add 6 — Inline magic-number comments (council R1 maintainability remediations)

- `.claude/hooks/session-start.sh` line counts (`head -5` for last-commit, `head -20` for active-plan).
- `.github/workflows/drift-check.yml` job timeout (`timeout-minutes: 5`).
- CLAUDE.md "Hook timeouts" section gets the `settings.json` timeout rationale (10s SessionStart / 15s PreToolUse) since JSON doesn't allow inline comments.

## Cost analysis (per the cost persona's R1 ask)

Pricing assumptions (Gemini, current as of V0.3.0 cut):
- **Pro:** ~$1.25 per 1M input tokens, ~$5 per 1M output tokens.
- **Flash:** ~$0.075 per 1M input, ~$0.30 per 1M output.
- **text-embedding-004:** ~$0.025 per 1M tokens.

Baseline scenario (per the cost persona's framing): **2 feature ticks/month + 10 PRs/month**.

| Surface | Per-invocation cost | Invocations/month | Monthly cost |
|---|---|---|---|
| Council on PR (existing) | ~$0.06 (6 personas × $0.01) | 10 | $0.60 |
| **+ maintainability persona** | +$0.01/PR | 10 | **+$0.10** |
| `harness research` | ~$0.08 (7 personas × $0.01) | 2 (one per feature tick) | **$0.16** |
| `harness synthesize --apply` | ~$0.005 (5 Flash × $0.001) | ≤1 (fires only on duplicate-signal clusters) | **<$0.005** |
| `harness reindex` | <$0.001 (corpus is small) | 1-2 (incremental) | **<$0.002** |
| **Total NEW dev-time cost** | | | **~$0.27/month** |

This is **well under** the existing ~$1–3/month council bill documented in CLAUDE.md. The marginal cost of V0.3.0 adoption at the baseline scenario is roughly the cost of one extra council-on-PR run per month.

**Headroom check:** at 5x baseline (10 feature ticks + 50 PRs/month), the V0.3.0-attributed surface costs ~$1.30/month. Still below the existing council baseline. The cost-cap mechanisms in `council.yml` (`MONTHLY_CAP`, cache-based counter) and `council.py` (`CALL_CAP=15` per run) continue to bound the council; the new commands are user-invoked and have no implicit triggers.

## Kill criteria (per the cost persona's R1 ask)

Each new dev-time AI surface gets explicit kill criteria. If any fires, disable the surface and re-evaluate.

- **`harness research`:** disable if (a) monthly research cost exceeds $5, OR (b) research output is ignored in 3 consecutive plans (verifies the signal is actually being used). Track via `failures.jsonl` entries when ignored. Disable by deleting the `research:` block from `harness.yml` (graceful degradation — `harness research` exits with "no config" error rather than silently using defaults).
- **`harness synthesize`:** disable if `--apply` is invoked on the same `signature_hash` more than once (idempotency failure). Disable by deleting the `synthesize:` block.
- **`harness recall` vector blend:** disable if reindex cost exceeds $1/month, OR if the vector blend's surfaced results are consistently less relevant than the keyword-only ranking (subjective, tracked in `learnings.md`). Disable by deleting `.harness/embeddings.json` (recall falls back to keyword-only).
- **`maintainability` persona:** disable if it consistently scores 9-10 with no actionable findings across 10 consecutive PRs (calibration noise — the persona's lens isn't catching anything new). Disable by renaming `.harness/council/maintainability.md` → `.harness/council/maintainability.md.disabled`.
- **drift-check workflow:** disable if it produces false-positive failures more than once per quarter, OR if `Anguijm/harness-cli` becomes unmaintained (no commits in 6 months). Disable by deleting the workflow file.

All kill criteria are reversible. None require code changes beyond config or file removal.

## devDependency-style justification for `Anguijm/harness-cli` (per the security persona's R1 ask)

Following CLAUDE.md's `devDependencies` rule for new dev-time external dependencies:

- **Maintainer / provenance:** John Anguiano (`@Anguijm` on GitHub). Same human who maintains this repo. Trust level = self-trust; this is not a third-party package.
- **Purpose:** Provides the canonical harness topology (workflows, hooks, persona templates, lib code). This repo's `harness check` (run weekly via `drift-check.yml`) compares this repo's `.harness/`, `.claude/`, `.github/workflows/` files against harness-cli's templates and reports drift.
- **Distribution model:** Public GitHub repo. No npm package, no published artifact. `drift-check.yml` clones at the pinned SHA via `actions/checkout` (SHA-pinned to the V0.3.0 release commit `519a3f5`).
- **Runtime impact:** **Zero.** harness-cli is dev-time only. Nothing in the consumer repo's runtime imports from harness-cli. The shipped `index.html` and Power Automate flows do not touch harness-cli. drift-check.yml runs in a CI sandbox and reports topology issues; it does not modify the consumer repo's working tree.
- **Update cadence:** When `Anguijm/harness-cli` cuts a new release tag, this repo bumps `HARNESS_VERSION` and `HARNESS_SHA` in a deliberate PR. No auto-bumping. SHA pinning protects against tag force-push.
- **Security review:** harness-cli's own `.github/workflows/council.yml` runs on every PR with the same multi-persona discipline. The V0.3.0 release passed council R3 with all scores 8-10.
- **Removal cost:** delete `.github/workflows/drift-check.yml`. Consumer repo loses weekly topology verification but suffers no functional regression.
